# 요구사항 정의

## 기능 요구사항

### FR-1. 이미지 입력 처리

- 시스템은 지정된 입력 루트 폴더 하위의 케이스 폴더들을 순회할 수 있어야 한다.
- 각 케이스 폴더에서 현재 이미지, 이전 정상 이미지, 1차 시각화 이미지를 식별할 수 있어야 한다.
- 필수 입력이 누락되었을 경우 오류 상태와 원인을 기록해야 한다.
- 기본 파일명 패턴은 아래 예시를 지원해야 한다.
- 이전 정상 이미지 예시: `2026-03-31T170358Z_SPOT.jpg`
- 현재 이미지 예시: `2026-04-01T105548Z_SPOT.jpg`
- 이상변경 시각화 이미지 예시: `dino_2026-04-01T105548Z_SPOT.png`
- 시각화 이미지는 현재 이미지 파일명을 기준으로 `dino_` prefix가 붙는 규칙을 1차 기준으로 사용한다.
- 입력 이미지 확장자는 `jpg`, `jpeg`, `png`를 모두 수용해야 한다.
- 개발용 샘플 입력으로 `test_images/<spot>/<case>/time_0.jpeg`, `time_n.png`, `dino_time_n.png` 구조도 지원해야 한다.
- 샘플 규약에서는 `time_0`를 이전 정상 이미지로, `dino_` prefix가 없는 나머지 단일 `time_*` 파일을 현재 이미지로 식별한다.
- 샘플 규약에서는 현재 이미지 파일명 앞에 `dino_`가 붙은 파일을 1차 이상 시각화 이미지로 식별한다.

### FR-2. 이미지 정렬 및 전처리

- 시스템은 이전 정상 이미지와 현재 이미지가 사전 정렬된 입력일 수 있음을 지원해야 하며, 필요 시 품질 점검 또는 경량 보정만 수행할 수 있어야 한다.
- 시스템은 VLM 입력 품질 향상을 위한 리사이즈, 크롭, 강조 영역 추출 등의 전처리를 수행할 수 있어야 한다.
- 시스템은 정렬 불가능 영역에 대해 생성된 검은색 마스크를 정보 없음 영역으로 취급하고 분석에서 제외해야 한다.
- 미세한 정렬 오차로 인한 소규모 위치 차이는 실제 변경으로 과대 판정하지 않아야 한다.
- 정렬 품질이 실질적 비교가 불가능할 정도로 낮을 때에만 해당 케이스는 `앵글 변경` 유형으로 처리하고 후속 이상 분석을 종료해야 한다.

### FR-3. 1차 이상 변경 결과 활용

- 시스템은 시각화 이미지의 파란색 관심 영역과 빨간색 이상 추정 패치를 해석 가능한 입력으로 변환해야 한다.
- 시스템은 전체 이미지 분석과 패치 중심 분석을 분리 또는 병행하여 수행할 수 있어야 한다.
- 기본 입력 전략은 ROI 크롭 이미지보다 1차 이상 변경 패치가 표시된 시각화 이미지 전체를 VLM에 함께 제공하는 것이다.
- ROI 크롭은 추후 옵션 확장 기능으로 둔다.

### FR-4. VLM 기반 이상 유형 분석

- 시스템은 현재 이미지, 이전 이미지, 패치/시각화 정보를 기반으로 이상 여부와 이상 유형을 판단해야 한다.
- 시스템은 위험성 기반 이상과 3S 기반 이상을 모두 고려해야 한다.
- 시스템은 반도체/제조 공장의 3S Patrol 관리자 관점에서 결과를 판단해야 한다.
- 시스템은 필요 시 다단계 프롬프팅 또는 결과 통합 로직을 사용할 수 있어야 한다.
- 일반적인 이상 여부 판단은 반드시 VLM 분석 절차를 통과해야 한다.
- rule-based 로직은 앵글 오류, 입력 이상, 비정상적 수집 실패를 식별하는 보조 판정에 한정한다.
- 아래 항목은 대표적인 실제 변경으로 우선 판정해야 한다.
- 장비/물체의 이동 (30cm 이상)
- 새로운 물체 등장
- 기존 물체 소실
- 액체 낙수/오염 발생
- 안전 테이프 파손/탈락
- 장비 커버 오픈/분리
- 타워램프 색상/상태 변화
- 화재 징후
- 낙상 위험
- 장비 파손
- 3S 기준 위반 (정리/정돈/청결)
- 안전 규정 위반
- 설비 이상 열화
- 아래 항목은 실제 변경 판단에서 기본적으로 제외해야 한다.
- 조도 변화 (그림자 길이/방향)
- 반사되는 빛, 빛 색깔 차이, 빛 번짐 현상
- 픽셀 단위 미세한 색상 차이
- 약간의 초점 불량
- 시간에 따른 조명의 변화
- 반사광 차이, 디스플레이 화면 출력 변경
- 먼지나 이물의 일시적 노출
- 검은색 마스크 영역의 변화
- 정렬 오차로 인한 미세 위치 차이
- 장비 창에 비친 내부 반영

### FR-5. 결과 통합

- 시스템은 전체 이미지 분석 결과와 패치 분석 결과를 통합하여 최종 결론을 생성해야 한다.
- 최종 결과에는 이상 여부, 이상 유형, 근거 요약, 신뢰도 또는 판단 상태가 포함되어야 한다.
- 최종 결과에는 아래 4개 핵심 산출물이 반드시 포함되어야 한다.
- 간단한 단어 중심 이상 변경 요약 설명
- VLM 이상 분석 진행 과정의 전체 텍스트 기록
- 분석 결과를 통합 정리하여 도출한 이상 검출 유형 내용
- 이상 검출 결과에 기반한 조치 권고 사항

### FR-6. 조치 권고 생성

- 시스템은 최종 이상 유형에 따라 상식적 수준의 초기 조치 권고를 생성해야 한다.
- 추후 별도 매뉴얼 문서를 입력받아 권고 체계를 고도화할 수 있어야 한다.

### FR-7. 로그 및 추적성

- 시스템은 VLM 분석 과정에서 생성된 텍스트 결과를 모두 저장해야 한다.
- 시스템은 주요 처리 단계별 성공/실패와 의사결정 근거를 기록해야 한다.

## 비기능 요구사항

### NFR-1. 모델 추상화

- 현재는 `GPT-5.4` 기반으로 개발/검증하되, 추후 `Kimi-K2.5`로 쉽게 교체 가능한 인터페이스를 유지해야 한다.

### NFR-2. 재현성

- 동일 입력과 동일 설정에서 가능한 한 일관된 결과를 재현할 수 있어야 한다.
- 평가 시 사용한 프롬프트, 파라미터, 모델 버전을 기록해야 한다.

### NFR-3. 확장성

- 새로운 이상 유형, 새 조치 규칙, 새로운 입력 메타데이터를 추가하기 쉬운 구조여야 한다.

### NFR-4. 관측 가능성

- 실패 지점, 누락 입력, VLM 응답 이상, 통합 단계 충돌을 추적할 수 있어야 한다.

### NFR-5. 운영 적합성

- 현장 담당자가 결과를 해석할 수 있는 간결한 출력 형식을 제공해야 한다.

### NFR-6. 표준 실행 환경

- Python 버전은 `3.11`을 기준으로 한다.
- 패키지 관리는 우선 `requirements.txt` 중심으로 운영한다.

## 입력 데이터 규약 초안

각 케이스 폴더는 아래와 같은 파일 집합을 가진다.

- `<timestamp>_SPOT.<jpg|jpeg|png>`: 이전 정상 이미지 또는 현재 이미지
- `dino_<current_timestamp>_SPOT.<jpg|jpeg|png>`: 현재 이미지에 대응하는 1차 이상 시각화 이미지

식별 규칙 초안:

- 원본 이미지 파일들 중 타임스탬프를 파싱할 수 있는 2개 파일에서 더 이전 시각의 파일을 이전 정상 이미지로 간주한다.
- 원본 이미지 파일들 중 타임스탬프를 파싱할 수 있는 2개 파일에서 더 이후 시각의 파일을 현재 이미지로 간주한다.
- 현재 이미지 파일명 앞에 `dino_`가 붙은 시각화 이미지 파일을 대응 결과로 간주한다.

개발용 샘플 규약:

- `test_images/<spot>/<case>/` 단위로 케이스를 구성한다.
- `time_0.<jpg|jpeg|png>`는 이전 정상 이미지다.
- `time_n.<jpg|jpeg|png>` 중 `n > 0`인 단일 파일은 현재 이미지다.
- `dino_time_n.<jpg|jpeg|png>`는 현재 이미지에 대응하는 시각화 이미지다.
- 예시: `test_images/spot_1/time_5/time_0.jpeg`, `time_5.png`, `dino_time_5.png`

## 출력 데이터 규약 초안

케이스별 필수 출력 항목:

- `abnormal_type`: 간단한 단어 중심 이상 변경 요약 설명
- `analysis_log`: VLM 이상 분석 진행 중 생성된 전체 텍스트 기록
- `abnormal_report`: 분석 결과를 통합 정리하여 도출한 이상 검출 유형 내용
- `action_guide`: 이상 검출 결과 기반 조치 권고 사항

권장 출력 포맷:

- 기계 처리용 `JSON`
- 사람 검토용 `Markdown` 리포트

권장 확장 필드:

- `case_id`
- `input_paths`
- `preprocess_summary`
- `status`
- `error_message`
- `model_info`
- `prompt_version`

예시:

```json
{
  "abnormal_type": "누수 의심",
  "analysis_log": [
    "reference image loaded",
    "current image loaded",
    "diff visualization reviewed",
    "VLM judged likely liquid leakage near floor"
  ],
  "abnormal_report": "이전 정상 이미지 대비 현재 이미지에서 바닥 주변 액체성 변화가 확인되며, 시각화 이미지의 표시 영역과 일치한다. 누수 또는 액체 유출 가능성이 있어 현장 확인이 필요하다.",
  "action_guide": "현장을 즉시 확인하고, 미끄럼 위험 구간을 통제한 뒤 누수 원인과 설비 이상 여부를 점검한다."
}
```

세부 스키마와 라벨 기준은 `docs/abnormal-taxonomy.md`를 기준 문서로 관리한다.

## 제약사항

- 현 단계에서는 실제 위험도 매뉴얼이 충분히 정리되어 있지 않다.
- 1차 시각화 이미지의 색상 규칙은 현재 설명 기준으로 고정 가정한다.
- 입력 이미지의 해상도, 각도 차이, 조명 차이 범위는 아직 명확하지 않다.
- 일반 케이스에서 단순 픽셀 차이만으로 이상 여부를 확정하지 않는다.
- 검은색 마스크 영역은 비교 근거로 사용하지 않는다.
- 반사광, 조명 변화, 디스플레이 출력 변화 같은 비구조적 시각 차이는 기본적으로 이상 근거에서 제외한다.
