# 확인이 필요한 항목

아래 항목들은 실제 구현 전에 구체화가 필요하거나 사용자 판단이 필요한 부분들이다.

## 1. 입력 데이터 규약

- 현재 기준 파일명 예시는 확정되었다.
- 이전 정상 이미지 예시: `2026-03-31T170358Z_SPOT.jpg`
- 현재 이미지 예시: `2026-04-01T105548Z_SPOT.jpg`
- 시각화 이미지 예시: `dino_2026-04-01T105548Z_SPOT.png`
- 이미지 확장자는 `jpg`, `jpeg`, `png`를 모두 수용하는 것으로 확정되었다.
- 개발용 샘플 입력은 `test_images/<spot>/<case>/time_0`, `time_n`, `dino_time_n` 규약으로도 제공되고 있다.
- 케이스별 메타데이터 파일도 함께 둘 예정인가?

## 2. 출력 스키마

- 최종 결과에는 아래 4개 내용이 필수다.
- `abnormal_type`: 간단한 단어 중심 이상 변경 요약 설명
- `analysis_log`: VLM 이상 분석 중 생성된 전체 텍스트 기록
- `abnormal_report`: 통합 정리된 이상 검출 유형 내용
- `action_guide`: 이상 검출 결과에 기반한 조치 권고 사항
- 사람 검토용 `Markdown` 리포트도 함께 생성한다.
- 신뢰도 점수 또는 위험도 등급을 포함할지?
- 이상 영역 좌표나 패치 위치 정보도 결과에 포함할지?

## 3. 이상 유형 taxonomy

- 기준 초안은 `docs/abnormal-taxonomy.md`로 분리했다.
- 위험 이상과 3S 이상을 하나의 라벨 체계로 유지할지, 상위 카테고리와 세부 라벨의 2단계 구조로 운영할지?
- `앵글 변경`, `정상`, `단순 변경`, `판단 불가`의 구분 기준을 더 엄격히 정의할지?

## 4. 조치 권고 수준

- 현재는 상식적 권고 수준으로 두었는데, 향후 어느 정도까지 구체화할지?
- 현장 조치 문구에 금지해야 하는 표현이 있는지?
- 보고 체계, 작업 중지, 접근 통제 같은 문구를 기본 포함해도 되는지?

## 5. 1차 시각화 이미지 해석

- 파란색/빨간색 표식의 색상 범위와 형식은 항상 동일한가?
- 빨간색 패치는 여러 개가 동시에 존재할 수 있는가?
- 패치 좌표를 프로그램적으로 추출할지, 단순 참조 이미지로만 사용할지?

## 6. 전처리/정렬 정책

- 비교용 이미지는 기본적으로 사전 기하학 정렬이 완료된 입력으로 받는다.
- 정렬 불가능한 가장자리 영역은 검은색 마스크로 제공되며, 해당 영역은 분석에서 제외한다.
- 미세한 정렬 오차는 허용하며, 실질적 비교가 불가능한 수준일 때만 `앵글 변경`으로 판단하고 종료한다.
- 기본 입력은 ROI 크롭보다 1차 이상 변경 시각화 이미지 전체를 함께 사용하는 방식으로 한다.
- ROI 기반 크롭 결과는 추후 옵션 기능으로 확장 가능하다.

## 7. 평가 하네스 기준

- 평가용 정답 라벨은 누가 작성할 예정인가?
- 하네스의 1차 성공 기준 수치는 어느 정도로 둘지?
- 설명/조치 품질 평가를 규칙 기반으로만 할지, LLM-as-Judge를 병행할지?

## 8. 실행 환경

- Python 버전은 `3.11`로 확정되었다.
- 패키지 관리는 우선 `requirements.txt` 기준으로 진행한다.
- 배치 실행 형식은 표준 CLI 형태로 진행한다.
- 예시: `python -m src.main --input-dir data/input --output-dir data/outputs`

## 9. 로그 및 추적성

- VLM raw response를 어느 수준까지 저장할지?
- 민감 정보 또는 사내 운영 정보 마스킹 정책이 필요한지?
- 실험 ID, 프롬프트 버전, 모델 파라미터를 어느 포맷으로 관리할지?

## 우선 답변 권장 항목

구현 착수 전에 특히 아래 항목들을 먼저 확정하는 것이 좋다.

1. 이상 유형 taxonomy의 계층 구조 세부화 여부
2. 평가 정답 데이터 형식
3. 신뢰도 점수 또는 위험도 등급 포함 여부
4. 이상 영역 좌표 저장 여부
5. 케이스별 메타데이터 파일 추가 여부
6. 로그 마스킹 정책 필요 여부
