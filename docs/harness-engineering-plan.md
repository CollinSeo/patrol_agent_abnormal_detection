# 하네스 엔지니어링 계획

## 목적

하네스 엔지니어링의 목적은 모델, 프롬프트, 전처리, 통합 로직이 바뀌더라도 동일한 데이터셋과 동일한 평가 절차로 성능과 품질을 반복 검증할 수 있게 만드는 것이다.

이 프로젝트에서는 하네스를 단순 테스트 스크립트가 아니라 아래 기능을 갖는 검증 프레임워크로 정의한다.

- 표준 입력 케이스셋 실행
- 기대 결과와 예측 결과 비교
- 실패 유형 분류
- 회귀 탐지
- 프롬프트/모델/파라미터 버전 기록

## 기본 원칙

- 하네스의 rule-based 평가는 일반 이상 판정의 대체물이 아니다.
- rule-based 평가는 입력 구조 검증, 파일 매칭, 정렬 실패 탐지, 극단적 앵글 오류 식별 같은 구조적 품질 점검에 우선 사용한다.
- 정상/이상 여부와 이상 유형 분류의 본 판단은 VLM 결과를 기준으로 평가한다.

## 하네스가 검증해야 할 대상

1. 입력 로딩 안정성
2. 전처리 및 정렬 성공 여부
3. 앵글 변경 케이스의 조기 종료 적절성
4. 패치 해석 결과 일관성
5. VLM 이상 유형 판단 품질
6. 결과 통합 로직의 일관성
7. 조치 권고 품질의 최소 타당성
8. 로그 저장 및 재현 가능성

## 평가 단위

기본 평가 단위는 케이스 폴더 1개다.

케이스별 입력:

- 현재 이미지
- 이전 정상 이미지
- 1차 시각화 이미지
- 필요 시 정답 메타데이터 파일

케이스별 출력:

- 구조화된 결과 JSON
- 원시 로그
- 사람 검토용 Markdown 리포트

## 권장 데이터셋 분류

### A. 정상 케이스

- 눈에 띄는 변화가 없거나 운영상 이상이 아닌 경우

### B. 명확한 위험 이상 케이스

- 화재, 연기, 누수, 파손 등 즉시 대응이 필요한 경우

### C. 3S 관리 이상 케이스

- 정리정돈 미흡, 장애물 적치, 오염 등 관리성 이상

### D. 혼동 유발 케이스

- 조명 변화, 촬영 각도 차이, 일시적 가림, 반사광 등

### F. 앵글 오류 케이스

- 정렬이 성립하지 않거나 잘못된 위치에서 수집되어 `앵글 변경`으로 종료되어야 하는 경우

### E. 경계 케이스

- 이상 여부가 애매하거나 사람 검토가 필요한 경우

## 정답 데이터 스키마 초안

케이스별 `expected.json` 예시:

```json
{
  "case_id": "case_001",
  "abnormal_detected": true,
  "abnormal_type": "누수/액체 유출",
  "severity": "medium",
  "must_mention": ["바닥", "액체", "미끄럼 위험"],
  "forbidden_labels": ["화재"],
  "recommended_action_keywords": ["현장 확인", "표지", "정리"]
}
```

## 평가 항목

### 1. 구조적 성공 여부

- 입력 파싱 성공
- 필수 출력 필드 생성
- 오류 없는 종료 여부

### 2. 앵글 변경 판정 적합성

- 정렬 실패 케이스에서 `앵글 변경`으로 종료하는지
- 앵글 오류가 아닌 케이스를 과도하게 탈락시키지 않는지

### 3. 이상 여부 판정 정확도

- `abnormal_detected` 정확도
- 정상/이상 이진 분류 정확도

### 4. 이상 유형 분류 적합성

- Top-1 유형 일치율
- 허용 동의어 매칭
- 상위 카테고리 기준 부분 점수

### 5. 설명 품질

- 필수 키워드 포함 여부
- 금지 라벨 미포함 여부
- 근거 설명의 이미지 기반성

### 6. 조치 권고 품질

- 유형과 조치 간 상식적 정합성
- 위험 유형에서 즉시 확인/통제 조치 포함 여부

### 7. 운영 품질

- 응답 누락 비율
- 재시도 횟수
- 처리 시간

## 자동 판정 방식 초안

### Rule-based 평가

- 입력 파일 구조, 파일 매칭, 타임스탬프 파싱, 정렬 실패, 앵글 오류를 평가
- VLM 출력 이후에는 이상 유형 라벨/동의어 집합 기준 비교를 수행
- 설명과 권고는 키워드 규칙 기반 1차 평가

### LLM-as-Judge 보조 평가

- 문장 품질과 조치 타당성은 보조적으로 LLM 판정을 사용할 수 있다.
- 단, 자동 회귀 기준은 가능하면 규칙 기반 점수를 우선 사용한다.

## 실험 축

하네스는 최소한 아래 변경 축을 비교 가능해야 한다.

- 모델 버전
- 프롬프트 버전
- 이미지 정렬 버전
- 패치 분석 사용 여부
- 통합 로직 버전
- 조치 권고 규칙 버전

## 실행 흐름 예시

1. 케이스셋 로드
2. 각 케이스에 대해 메인 파이프라인 실행
3. 결과를 JSON으로 저장
4. 기대값과 비교 평가
5. 실패 케이스와 요약 통계 생성
6. 이전 실험 결과와 비교해 회귀 여부 판정

## 결과 리포트 예시 항목

- 실행 ID
- 데이터셋 버전
- 모델/프롬프트 버전
- 총 케이스 수
- 정상/이상 정확도
- 유형 분류 정확도
- 권고 품질 통과율
- 실패 상위 유형
- 대표 실패 사례 링크

## 개발 단계별 하네스 적용 계획

### 1단계. 골격 구축

- 입력 케이스 구조 확정
- 결과 JSON 스키마 확정
- 최소 5~10개 샘플 케이스 수집
- Markdown 리포트 템플릿 확정

### 2단계. 기준선 수립

- 단일 프롬프트 기반 분석 파이프라인 구현
- `앵글 변경` 종료 기준 구현
- 초기 정확도 측정

### 3단계. 비교 실험

- 패치 사용 여부, 정렬 여부, 프롬프트 버전 비교
- 실패 유형 분석

### 4단계. 회귀 관리

- 코드/프롬프트 변경 시 정기 하네스 실행
- 기준선 대비 저하 탐지

## 합격 기준 초안

정량 기준은 아직 확정 전이지만 초기에는 다음과 같은 게이트를 둘 수 있다.

- 필수 출력 필드 생성율 100%
- 치명적 실행 실패율 0%
- 정렬 실패 케이스의 `앵글 변경` 라우팅 정확성 검토
- 이상 여부 판정 정확도 목표 설정 필요
- 위험 이상 케이스에서 조치 권고 누락 0%

정확한 수치는 데이터셋 규모와 난이도 확인 후 확정한다.
